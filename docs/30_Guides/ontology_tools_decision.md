# 온톨로지 도구(Fuseki/Protégé) 미도입 사유 및 시스템 설계 철학

본 문서는 온톨로지 구축 및 관리에 널리 사용되는 도구인 Apache Jena Fuseki와 Protégé를 현재 시스템(COA Agent Platform)에 도입하지 않기로 결정한 기술적 배경과 사유를 설명합니다.

## 1. 도구 개요

### 🔧 Apache Jena Fuseki
*   **정의**: SPARQL 서버로서 동작하는 RDF 트리플 스토어(Triple Store).
*   **주요 기능**:
    *   RDF 데이터를 저장하고 HTTP를 통해 SPARQL 쿼리 엔드포인트를 제공.
    *   대규모 RDF 데이터의 인덱싱 및 검색 지원.
    *   웹 기반 UI를 통한 데이터 관리 및 쿼리 테스트.

### 🎨 Protégé
*   **정의**: 스탠포드 대학에서 개발한 오픈소스 온톨로지 편집기.
*   **주요 기능**:
    *   GUI 기반의 온톨로지 클래스, 속성, 인스턴스 생성 및 편집.
    *   OWL/RDF 형식의 데이터 시각화 및 검증.
    *   추론기(Reasoner) 플러그인을 통한 정합성 검사.

---

## 2. 현재 시스템의 접근 방식

현재 COA Agent Platform은 위 도구들을 사용하는 대신, **Python 기반의 경량화된 파이프라인**을 자체 구현하여 운영하고 있습니다.

*   **Excel 기반 데이터 관리**: 현업 담당자에게 친숙한 Excel 포맷으로 원천 데이터를 관리.
*   **자동 변환 파이프라인**: `EnhancedOntologyManager`가 Excel 데이터를 읽어 RDF/TTL 파일로 자동 변환.
*   **In-Memory 처리**: `rdflib` 라이브러리를 사용하여 Python 메모리 상에서 그래프를 로드하고 SPARQL 쿼리를 수행.

## 3. 미도입 사유 (기술적 판단)

### ✅ 시스템 효율성 및 구조적 단순함
*   **Excel → TTL 자동화**: 데이터가 변경될 때마다 Protégé에서 수동으로 편집하는 것은 비효율적입니다. 현재 시스템은 코드 레벨에서 변환 로직을 수행하므로 데이터 업데이트 속도가 매우 빠릅니다.
*   **배포 용이성**: Fuseki와 같은 별도의 Java 기반 서버를 띄우고 관리하는 것은 시스템 복잡도를 불필요하게 증가시킵니다. 현재는 Python 런타임 하나로 모든 기능이 완결됩니다.

### 📉 데이터 규모 적합성
*   **중소형 규모**: 현재 다루는 전장 정보와 교리 데이터는 수천~수만 트리플 수준입니다. 이는 `rdflib`를 사용하여 인메모리에서 처리하기에 충분한 크기이며, 별도의 DB 서버(Fuseki)가 필요한 'Big Data' 수준이 아닙니다.

### 👤 단일 사용자/단일 서버 환경
*   **동시성 이슈 없음**: 현재 시스템은 분석가가 단독으로 시뮬레이션을 수행하거나, 단일 서버에서 동작하는 구조입니다. 여러 사용자가 동시에 트리플 스토어에 접근해 트랜잭션을 일으키는 상황이 아니므로 SPARQL 서버가 불필요합니다.

---

## 4. 향후 도입 고려 조건

다음과 같은 상황이 발생할 경우, Fuseki 또는 Protégé의 도입을 재검토할 수 있습니다.

1.  **데이터 규모 폭증**: 인스턴스 데이터가 수십만~수백만 개를 넘어서 메모리 처리가 불가능해질 때.
2.  **실시간 다중 접속**: 여러 부서나 분석가가 동시에 온톨로지 데이터를 수정하고 조회해야 하는 협업 환경이 될 때.
3.  **고도화된 추론 필요**: 단순 SPARQL 쿼리를 넘어, OWL DL 수준의 복잡한 논리적 추론(Reasoning)이 실시간으로 요구될 때.
